========================================
STRICT CONVERSATION TURN DISCIPLINE

The backend must:

* Return exactly ONE assistant_message per request.
* Never generate two assistant messages in one cycle.
* Never generate a confirmation summary AND a booking confirmation in the same response.
* Never suppress a Claude-generated message accidentally.
* Never overwrite assistant_message after it is set.

If book_now == true:

* Claude must set assistant_message = "" (empty string).
* Backend will generate the final confirmation message itself.
* Only one message is returned to the client.

========================================
DATE + TIME HANDLING (CRITICAL)

Use server current date (datetime.utcnow()) as ground truth.

All natural language dates must be interpreted relative to current server date.

Never hallucinate future months.

If ambiguity exists (e.g., unclear morning vs afternoon), ask clarification instead of guessing.

Convert preferred_date + preferred_time into timezone-aware UTC datetime.

Store internally in ISO 8601 format.

All appointments are 60 minutes.

Examples:

* “tomorrow morning” → next calendar day
* “next Monday” → next upcoming Monday
* “this Friday” → upcoming Friday (if future), not past

========================================
APPOINTMENT RULES

All appointments are 60 minutes.

Prevent overlapping time windows per session.

Conflict detection:

* Convert preferred_date + preferred_time to datetime.
* Block scheduling if overlaps with any existing appointment in scheduled_appointments[session_id].

========================================
SCHEDULING FLOW (TWO-STEP CONFIRMATION)

STEP 1 — Intake Phase

Claude collects:

* full_name
* contact_info
* reason_for_visit
* visit_category
* preferred_date
* preferred_time

When all required fields are present:

* Claude sets confirm_ready = true
* Claude sets book_now = false
* Claude generates assistant_message summarizing details

Example:

"Here’s what I have:

Name: John Smith
Visit: General Consultation
Date: 2026-02-27
Time: 09:00

Shall I book this?"

Backend does NOT schedule yet.

STEP 2 — User Confirmation

If user responds affirmatively AND Claude sets:

* book_now = true
* assistant_message = ""

Then backend must:

* Validate no emergency.
* Validate no conflict.
* Call schedule_appointment() stub synchronously.
* Store ScheduledAppointment in scheduled_appointments.
* Generate final confirmation message.

Example:

"Done! Your General Consultation appointment on February 27 at 9:00 AM is confirmed (ID: 48291). Is there anything else I can help you with?"

* Generate calendar_event object.

* Clear from ConversationState:

  * reason_for_visit
  * visit_category
  * preferred_date
  * preferred_time
  * confirm_ready
  * book_now

* RETAIN:

  * full_name
  * contact_info

This allows user to book another appointment without re-entering identity.

========================================
EMERGENCY HANDLING

If is_emergency == true:

* Immediately return emergency instruction message.
* Do NOT continue scheduling.
* Do NOT call schedule_appointment.
* Do NOT modify appointment lists.
