# BACKEND CONTRACT

The backend already exists in:

```
../backend
```

You may read it for reference.
You may NOT modify it.

API:

POST /chat

Request:

```
{
  "session_id": String,
  "message": String
}
```

Response:

```
{
  "session_id": String,
  "assistant_message": String
}
```

Important:

* Session state lives entirely server-side.
* The client sends exactly one message at a time.
* The backend will always return exactly one assistant_message.
* The backend will not stream responses.
* The backend will not return multiple assistant messages in one response.

Therefore:

Chat history will always be strictly alternating:
User → Assistant → User → Assistant

Do NOT resend message history.

Do NOT coalesce messages.

* Disable Send button immediately.

  * Typing animation completes

Keyboard behavior:

* Enter → Send message
* Shift + Enter → Insert newline
* Cmd + Enter (Mac Web) → Insert newline
* Send only triggers when plain Enter is pressed without modifiers

No message coalescing logic is needed.

# NETWORKING

Use dio with CancelToken.

Flow:

When user presses Send:

1. Append user message immediately.
2. Disable input + send.
3. Fire POST request.

If request fails:

* Insert error bubble:
  "Something went wrong. Tap to retry."
* Re-enable send.
* Retry removes error bubble and resends last user message.

No exponential backoff.

---

# TYPING SIMULATION (VERY IMPORTANT)

When user sends a message:

1. Start fixed 650ms "reading delay" timer.
2. Fire network request immediately in parallel.

If response arrives before 650ms:

* Cancel delay.
* Do NOT show typing indicator.
* Immediately begin character animation.

If 650ms passes before response:

* Show typing indicator (animated three dots).
* Indicator remains visible until character animation begins.

When response arrives:

* Remove typing indicator.
* Insert AI message bubble immediately.
* Animate text character-by-character.
